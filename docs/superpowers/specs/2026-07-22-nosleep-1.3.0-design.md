# nosleep 1.3.0 design: low-battery refusal + lid-close chime

Date: 2026-07-22
Status: approved (design), pending spec review

Two changes ship together in 1.3.0 because they touch the same plumbing (the
root background watcher's argument list and the enable code paths):

1. **Low-battery refusal + `insist`** (bugfix): stop showing a misleading "No
   Sleep ON" banner when the battery is below the cutoff and the watcher is
   about to turn it right back off. Refuse up front; add `insist` to override.
2. **Lid-close chime** (feature): play a short system sound when the lid closes
   while nosleep is on, as audible confirmation it is holding the Mac awake.

Shared change: the watcher's internal entrypoint grows from
`__watch <deadline>` to `__watch <deadline> <batt_min> <sound>`.

---

## Feature A: low-battery refusal + `insist`

### The bug

`set_on` / `set_timed` run `sudo pmset -a disablesleep 1`, print the red "No
Sleep ON" owl, then launch the detached root watcher. On its first poll the
watcher sees `battery_pct < BATTERY_MIN` (15%) and immediately runs
`pmset -a disablesleep 0`. Result: a confident "ON" banner for a state that is
already off within moments.

### New behavior

Before any enable path touches `pmset`, run an enable-time battery guard. If the
machine has a battery, its charge is below `BATTERY_MIN`, and the user did not
pass `insist`, refuse without flipping anything:

```
Battery <15%. nosleep not enabled.
'nosleep 30m insist' to turn on anyway.
```

The second line is dynamic and mirrors the invocation (modifier form):

| You ran                | Hint shown                                  |
|------------------------|---------------------------------------------|
| `nosleep` / `nosleep on` | `'nosleep insist' to turn on anyway.`     |
| `nosleep 30m`          | `'nosleep 30m insist' to turn on anyway.`   |
| `nosleep 1h clean`     | `'nosleep 1h insist clean' to turn on anyway.` |

`15` in the message is `BATTERY_MIN`, not a literal. The message is plain
(yellow) text in the style of the existing "Aborted." messages, not an owl.

**Placement matters:** the guard runs at dispatch, *before* `do_clean` quits any
apps. Otherwise a low battery would quit all your apps and then refuse. So the
guard sits ahead of `do_clean`, not inside `set_on`/`set_timed`.

### `insist` (modifier)

`insist` is a boolean modifier parsed like `clean`. It attaches to any enable
action and preserves the original intent:

- `nosleep insist` (bare / `on`): force on indefinitely.
- `nosleep 30m insist`: force on for 30 minutes.
- `nosleep insist clean` / `nosleep 1h insist clean`: clean, then force on.

`insist` is rejected (usage error) when combined with `off`, `status`, or
`whitelist` (and `sound`, which is intercepted earlier). `help` and `version`
are terminal informational commands: they print and exit before validation
runs, so `insist` alongside them is a harmless no-op rather than an error
(they win, like `--help`/`--version` in most CLIs).

**`insist` disables the battery safety for that session.** Bypassing only the
enable-time check would be pointless: the watcher's own 15% cutoff would turn it
off within ~30s. So `insist` passes an effective battery floor of `0` to the
watcher (see shared plumbing), which never trips. When insisting, the banner
replaces the usual "Will restore sleep if battery drops below 15%" note with a
warning that the cutoff is **off** for this session, so the user knows the
protection they normally have is disabled.

### Affected functions (Feature A)

- New `battery_guard <suggestion-cmd>`: the refusal check; `exit 1` on refuse.
- New helper to reconstruct the suggestion string from parsed `DUR`/`CLEAN`
  (keep the original duration *token* like `30m`, not the parsed seconds, so a
  new `DUR_TOKEN` is captured during parsing).
- Dispatch: call `battery_guard` in the `on`, duration, `clean`, and
  toggle-when-off branches (before `do_clean`).
- `EFFECTIVE_BATT_MIN` global: `0` when `INSIST=1`, else `BATTERY_MIN`. Passed
  into `start_watcher` / the watcher.
- `battery_note`: insist-aware wording.

## Feature B: lid-close chime

### Behavior

On a laptop, while nosleep is on, closing the lid plays a short system sound.
Opening is silent. Fires on every close. On by default; a laptop running nosleep
already has the watcher running, which is where the detection lives.

### The `sound` subcommand

Handled early in dispatch (like `__watch`), so it does not tangle with the
regular parser: `if [ "${1:-}" = sound ]; then shift; cmd_sound "$@"; exit; fi`.

- `nosleep sound`         : show the current setting and list available sound names.
- `nosleep sound off`     : silence the chime (write `off`).
- `nosleep sound on`      : re-enable with the default sound.
- `nosleep sound <Name>`  : validate the name exists in
  `/System/Library/Sounds/<Name>.aiff`, play a one-shot **preview** (we are
  running as the user here, so a direct `afplay` works, no `launchctl` needed),
  then persist. Unknown name: error listing the valid names.

### Config

- New file `$CONFIG_DIR/sound` (alongside the existing `whitelist`).
- Contents: a sound name (e.g. `Submarine`) or the literal `off`. Absent file =
  default sound, on.
- `DEFAULT_SOUND="Submarine"` (a distinctive sonar "boop"; noticeable, not
  alarming). Overridable anytime via `nosleep sound <Name>`.
- `load_sound`: read the file; return `off`, a valid name, or the default. If a
  stored name's `.aiff` is missing, fall back to the default.

### Watcher: adaptive two-speed polling

The chime detection lives in `run_watch`. It tracks the previous lid state and
chimes only on an open->closed transition. Polling cadence is adaptive:

- **Lid open** (user likely nearby, about to close): fast-poll
  `AppleClamshellState` every `LID_POLL` (~2s) so the chime lands promptly.
- **Lid closed** (chimed, walked away): drop back to `POLL_INTERVAL` (~30s). We
  only need to eventually notice the *reopen* to re-arm; there is no latency
  requirement because opening is silent. On reopen, speed back to 2s.

So the 2s polling runs only while the lid is open, which is exactly when the
machine is in use. Battery/deadline checks stay gated to ~30s regardless of tick
speed (via an elapsed-time check, so fast lid ticks do not run `pmset -g batt`
every 2s).

When the sound is `off`, `run_watch` behaves exactly as today: a 30s loop with
no lid polling. This preserves the current low-power profile for anyone who
disables the chime.

**Lid detection:**
```sh
lid_closed() { ioreg -r -k AppleClamshellState -d 4 | grep -q '"AppleClamshellState" = Yes'; }
```

### Playing sound from root into the user's audio session

The watcher runs as root, and root cannot emit audio into the logged-in user's
session directly. Route through the console user:

```sh
play_sound() {                       # $1 = sound name
  local f="/System/Library/Sounds/$1.aiff"
  [ -f "$f" ] || return 0
  local user uid
  user=$(stat -f%Su /dev/console 2>/dev/null) || return 0
  [ -z "$user" ] || [ "$user" = root ] && return 0   # no GUI user / loginwindow
  uid=$(id -u "$user" 2>/dev/null) || return 0
  launchctl asuser "$uid" sudo -u "$user" afplay "$f" >/dev/null 2>&1 &
}
```

Best-effort and backgrounded: no GUI user, a missing file, or a play failure
degrades to silence and never disrupts the watcher's real job (battery/timer
auto-off). `sudo -u` needs no password because the watcher is already root.

### Accepted edge

If the user reopens and re-closes within the ~30s slow window (before the
watcher has re-armed), the chime may be missed on that quick re-close. Given the
"close and walk away" use case, this is an acceptable trade for keeping the loop
cheap. Documented in the README.

## Shared plumbing

### New watcher signature

`__watch <deadline> <batt_min> <sound>`

- `deadline`: unchanged (epoch seconds; `0` = never).
- `batt_min`: effective battery floor (`15` normally, `0` when insisted).
- `sound`: sound name, or `off`.

`run_watch(deadline, batt_min, sound)` uses `batt_min` in its cutoff test
(replacing the hardcoded `BATTERY_MIN`) and `sound` to decide whether to
lid-poll and what to play.

`start_watcher` gains the `batt_min` and `sound` values (resolved in user
context before `sudo`, because `sudo` scrubs the environment) and bakes them
into the launch args. It still skips launching only when there is genuinely
nothing to do: `deadline = 0` AND `batt_min = 0` AND `sound = off`.

### Parser changes

- `insist` keyword -> `INSIST=1`; validated against incompatible commands.
- `DUR_TOKEN` captured alongside `DUR` for the refusal hint.
- `sound` intercepted before the main parse loop.

## Testing

- `shellcheck nosleep` clean.
- **Low-battery refusal**: temporarily raise `BATTERY_MIN` (or mock
  `battery_pct`) so the guard trips at the current charge. Verify: the exact
  two-line message, the dynamic hint for `on` / `30m` / `1h clean`, that
  `pmset` state is unchanged (still off), and that `clean` refuses *before*
  quitting apps. Then `insist`: verify it enables, shows the "safety off" note,
  and passes `batt_min = 0` to the watcher (state stays on).
- **Chime**: `nosleep on`, close lid -> Submarine within ~2s (verify audible
  with lid physically shut); reopen -> silent. `nosleep sound off` -> close ->
  silent. `nosleep sound Glass` -> preview plays + persists -> close -> Glass.
  `nosleep sound` -> shows setting + list.
- **Regression**: timer auto-off and the normal 15% cutoff (non-insist) still
  fire; `status` unchanged; desktop (no battery/lid) behavior unchanged.

## Out of scope (YAGNI)

- Lid-*open* chime, or a distinct auto-off ("about to sleep") sound.
- Bundled custom audio assets; we use stock `/System/Library/Sounds` only.
- Any GUI / menu-bar surface.
- Making the 2s latency or the poll cadence user-configurable.
